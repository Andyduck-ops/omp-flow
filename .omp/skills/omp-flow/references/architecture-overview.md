# OMP-Flow Architecture Overview

omp-flow is a multi-agent workflow orchestration framework that combines Trellis-style context management with Maestro-style Ralph FSM for structured, verifiable, and self-iterating agent workflows inside a project.

This document describes the system model from the perspective of someone reading and modifying the `.omp-flow/` workspace -- not the upstream source code.

---

## 1. File Inventory with Editability Markers

The `.omp-flow/` directory tree is the heart of omp-flow's state. Files fall into three editability categories:

### Runtime State (DO NOT EDIT BY HAND)

These files are written programmatically by the omp-flow runtime. Manual edits will be overwritten or cause desynchronization.

| Path | Content | Written By |
|------|---------|------------|
| `.omp-flow/state.json` | Workspace state: milestone, phase, FSM state, active wave, goals, tasks, spec rules | `UnifiedWorkspaceManager`, `RalphFSMEngine.transitionTo()` |
| `.omp-flow/fsm/ralph-<id>/status.json` | FSM session: steps, decision log, boundary contract, task decomposition | `RalphFSMEngine.saveStatus()` |
| `.omp-flow/fsm/ralph-<id>/decisions.ndjson` | Append-only decision log entries | `RalphFSMEngine.appendDecisionToFile()` |
| `.omp-flow/events/events.jsonl` | Append-only event stream (19 event types) | `EventBus.append()` |
| `.omp-flow/events/events.jsonl.seq` | Monotonic sequence counter for event dedup | `EventBus.nextSequence()` |
| `.omp-flow/events/discoveries.ndjson` | Cross-agent discovery board | `EventBus.appendDiscovery()` |

### User-Editable Configuration (EDIT FREELY)

These files define project-specific omp-flow behavior. They are created by `init` but intended for modification.

| Path | Content | Purpose |
|------|---------|---------|
| `.omp-flow/specs/*.md` | Spec rule files for coding conventions, architecture decisions | Referenced by subagents via agent pull |
| `.omp-flow/knowhow/harvested-learnings.md` | Harvested patterns, gotchas, and architectural insights | Accumulated learnings from past work |
| `.omp-flow/sessions/` | Per-session scratch data | Temporary analysis artifacts |
| `.omp/skills/omp-flow-local/SKILL.md` | Project-specific skill customizations | Extensions and overrides to core omp-flow |

### Semi-Automated (S3: Generated, Optionally Editable)

These files are generated by omp-flow tools but can be reviewed and refined by the user.

| Path | Content | Generator |
|------|---------|-----------|
| `.omp-flow/tasks/*/prd.md` | Product requirements document | `omp-flow-architect` skill |
| `.omp-flow/tasks/*/design.md` | Design decisions | `omp-flow-architect` skill (optional) |
| `.omp-flow/tasks/*/research/*.md` | Domain research | `omp-flow-researcher` skill |
| `.omp-flow/tasks/*/brainstorm.md` | Brainstorming output | `omp-flow-brainstorm` skill |
| `.omp-flow/findings/*.md` | Audit findings | `omp-flow-reviewer` / `omp-flow-debugger` |

## 2. Core Paths Map

```
.omp-flow/
+-- state.json                        # Workspace state (milestone, phase, FSM state, goals)
+-- tasks/                            # Task artifacts
|   +-- .active-task                  # Current task slug pointer
|   +-- {taskId}/                     # Per-task directory
|       +-- prd.md                    # Requirements document
|       +-- design.md                 # Design decisions
|       +-- implement.md              # Implementation plan
|       +-- brainstorm.md             # Brainstorming output
|       +-- research/                 # Domain research files
|       +-- analysis/                 # Analysis artifacts
|       +-- *.jsonl                   # Context manifests
|       +-- *.csv                     # Structured data
+-- specs/                            # Spec rules (coding conventions)
|   +-- *.md                          # Per-category spec files
+-- knowhow/                          # Harvested learnings
|   +-- harvested-learnings.md        # Accumulated patterns and gotchas
+-- fsm/                              # Ralph FSM state
|   +-- ralph-{sessionId}/
|       +-- status.json               # Session step state, decision log
|       +-- decisions.ndjson          # Append-only decision entries
+-- events/                           # Event system
|   +-- events.jsonl                  # Append-only event stream
|   +-- events.jsonl.seq              # Sequence counter
|   +-- discoveries.ndjson            # Cross-agent discovery board
+-- findings/                         # Audit findings
|   +-- *.md                          # Per-pass findings
+-- sessions/                         # Per-session data
|   +-- {sessionId}/                  # Session scratch directory
+-- scratch/                          # Transient working files
+-- issues/                           # Issue tracking
```

### Key Symlinks and Pointers

| Pointer | Format | Purpose |
|---------|--------|---------|
| `.omp-flow/tasks/.active-task` | Single line: `{taskSlug}` | Points to the currently active task |
| `.omp-flow/fsm/ralph-*/` | Directory per session | Active FSM sessions (may be multiple) |

## 4. AI Customization Principles

Derived from the patterns observed across Trellis and omp-flow codebases. These principles guide safe, maintainable modifications to omp-flow inside a user project.

### Principle 1: Find the local source of truth first

Do not edit from memory. Read `.omp-flow/state.json`, `.omp-flow/specs/`, `.omp-flow/fsm/ralph-*/status.json`, and the relevant task files first. The file system is the source of truth for what omp-flow is doing.

### Principle 2: Edit the user project, not the upstream source

Modify files inside the project's `.omp-flow/` and `.omp/skills/` directories. Do not modify `node_modules/`, the OMP npm package cache, or cloned framework repositories unless the user explicitly wants to contribute upstream.

### Principle 3: Keep spec files aligned with FSM behavior

If spec rules or workflow routing changes, verify that the FSM state transitions, skill routing in `status.json`, and skill definitions still describe the same flow. A mismatch causes agents to act on stale instructions.

### Principle 4: Put project-specific conventions in local skills, not core

Do not put team-specific or project-specific rules into `.omp/skills/omp-flow/SKILL.md` (the core skill). Instead, use `.omp/skills/omp-flow-local/SKILL.md` for project customizations. This keeps `omp-flow` upgrade-safe and prevents merge conflicts during updates.

### Principle 5: Preserve user changes and accumulated data

If a file was already modified locally (spec files, knowhow, task PRDs), work from the current content instead of overwriting it with defaults. Append to `events.jsonl` and `discoveries.ndjson` -- never delete or replace them. The event stream is append-only by design.
